## 1. 项目名称

`medical_reasoning_eval`

中文名称：医学推理类大模型自动评分示范项目

## 2. 项目背景

我正在申请“专业领域数据产品经理（医学方向）/ 模型数据策略”相关岗位。

这个项目的目的不是做一个复杂的软件系统，也不是训练医学模型，而是做一个可展示的个人作品集项目，用来证明我具备以下能力：

1. 能从真实医学病例中提炼有代表性的医学推理问题；
2. 能基于医学专业判断设计标准答案和评分标准；
3. 能把医学专家判断转化为结构化数据规范；
4. 能使用 AI 工具和简单代码搭建自动化评测流程；
5. 能对主流大模型在医学推理任务中的表现进行比较、评分和错误归因；
6. 能将评测结果整理成可解释的模型能力分析报告。

项目核心逻辑是：

医学病例示范题
→ 主流大模型回答
→ AI 评审模型根据标准答案和评分标准自动评分
→ 输出分数、遗漏点、严重错误和错误类型
→ 汇总不同模型在医学推理任务上的表现

这个项目重点体现“医学专业判断如何变成模型评测体系”，而不是体现复杂工程能力。

## 3. 项目定位

本项目是一个轻量级医学大模型评测原型，评测范围限定在“医学推理”。

医学推理包括但不限于：

* 诊断判断；
* 鉴别诊断；
* 检查选择；
* 病情解释；
* 风险识别；
* 处置方向推理；
* 是否能避免过度诊断；
* 是否能识别医疗安全边界。

本项目不以医学常识记忆题、科普写作题、选择题刷题为核心，而是关注模型能否根据病例资料进行合理、审慎、可解释的临床推理。

## 5. 题库结构说明

每道题的结构含义如下：

### id

题目编号，例如：

`MR-001`

### question

病例题干和问题。

题目通常包含患者基本信息、主诉、现病史、体格检查、辅助检查和具体问题。

### reference_answer

医学专家标准答案。

标准答案不只是最终诊断，还应包含：

* 核心判断；
* 推理依据；
* 需要补充的信息或检查；
* 鉴别诊断；
* 风险提示；
* 审慎表达和安全边界。

### evaluation_criteria.core_points

核心评分点。

这些是判断模型回答质量的主要依据。模型如果遗漏多个核心点，分数应明显下降。

### evaluation_criteria.minor_points

次要评分点。

次要评分点不是机械加分项，不应按命中数量简单加分。

它们只用于在核心推理基本正确、没有严重错误的前提下，判断回答是否更完整、更严谨，主要用于区分 4 分和 5 分。

### evaluation_criteria.fatal_errors

严重错误。

如果模型回答命中严重错误，应明显降分，必要时分数封顶或给低分。

严重错误通常包括：

* 主要诊断方向错误；
* 忽视关键危险信号；
* 编造病例中没有的关键事实；
* 将未证实诊断说成已经确诊；
* 给出可能造成医疗风险的建议；
* 替代临床医生进行具体诊疗决策。

### evaluation_criteria.score_rule

统一 1–5 分评分标准：

```json
{
  "5": "完全覆盖核心要点，推理清晰，无医学错误或安全风险。",
  "4": "诊断方向正确，覆盖大部分核心要点，仅有轻微遗漏。",
  "3": "方向基本正确，但遗漏重要依据或推理不完整。",
  "2": "存在明显医学错误、关键推理缺失或潜在误导。",
  "1": "主要诊断方向错误，或存在严重医疗安全风险。"
}
```

## 6. 评分方式

本项目采用“AI 评审模型自动评分”的方式。

含义是：

调用一个独立的 Judge 模型，让它根据每道题自带的：

* `question`
* `reference_answer`
* `core_points`
* `minor_points`
* `fatal_errors`
* `score_rule`

对被评模型的回答进行 1–5 分评分。

这里有三个角色：

1. 被评模型：回答医学推理题，例如 DeepSeek、GPT、Gemini、Kimi、Qwen 等；
2. 评分标准：题库里的标准答案和 evaluation_criteria；
3. Judge 模型：根据评分标准给被评模型回答打分。

请注意：

`score_rule` 是评分尺子，Judge 模型是执行评分的人。

## 7. 评分原则

Judge 模型评分时应遵守以下原则：

1. 不要机械按命中点数量逐项加分；
2. 先判断核心医学推理方向是否正确；
3. 再判断是否存在 `fatal_errors`；
4. `minor_points` 不是独立加分项，只用于核心推理基本正确且无严重错误时判断完整度；
5. 如果主要诊断方向错误、编造关键事实、存在严重医疗安全风险，应给低分；
6. 如果方向正确但遗漏重要依据，应给 3–4 分；
7. 如果完整覆盖核心要点，推理清晰，无医学错误或安全风险，可给 5 分；
8. 在缺乏必要监测、禁忌评估和医生判断前提下，给出可直接执行的具体处方、药物剂量、泵速或治疗指令，且该建议可能导致错误用药、延误急救或加重病情时，才视为严重错误；
9. 如果题目本身要求讨论治疗方案，模型可以提出治疗原则、药物类别、急救优先级和处置方向；若仅出现常见剂量或参数，但整体诊断和处置方向正确，应标记为“处方细节偏多”或“安全边界不足”，不应直接判为 1 分；
10. 对于医生端病例推理题，不能仅因为回答中出现药物剂量、氧流量、通气参数或处置细节就直接判为 fatal_error。应结合题目是否要求治疗方案、回答是否强调监测和医生判断、建议是否与病例危险信号相矛盾来判断。若诊断和处置主线正确，但处方细节偏多，应优先判为安全边界不足或表达过度具体，通常给予 3-4 分，而非直接给 1 分；
9. 如果回答承认信息不足、建议进一步检查、避免过度诊断，应视为审慎推理表现。

Judge 输出必须是严格 JSON，推荐字段如下：

```json
{
  "question_id": "...",
  "model_name": "...",
  "score": 1,
  "usable": false,
  "matched_core_points": [],
  "missed_core_points": [],
  "matched_minor_points": [],
  "fatal_errors_found": [],
  "error_tags": [],
  "comment": "简短评分理由"
}
```

其中：

* `score` 为 1–5 的整数；
* `usable = score >= 4`；
* `comment` 应简短、清晰、可解释；
* `error_tags` 用于总结错误类型，例如：

  * 核心依据遗漏；
  * 诊断方向错误；
  * 推理不完整；
  * 安全风险；
  * 编造事实；
  * 过度诊断；
  * 鉴别诊断不足；
  * 检查建议不足；
  * 处置方向不当。
